1. 디자인 톤 자체가 약간 “스타트업 포트폴리오” 감성

네온색, 다크모드, 아이콘 잔뜩, Urbanist 폰트
→ 개발자/디자이너 발표용으로는 예쁜데,
→ 팀장·임원 기준으론 “멋있네?” 보다는 “뭘 얼마나 했는데?”가 먼저라서, 디자인 인상이 내용보다 눈에 많이 들어옴.

첫 슬라이드도 “MS Graph API 연동 및 아웃룩 자동화 체계 수립” 이런 문구가 기술 중심.
→ 회사 중간보고라면

“무분별한 공지 메일을 통제하고, 승인/기록이 남는 공지 체계 구축”
같은 업무/통제 관점 문장이 먼저 오는 게 더 조직스러움.

정리:
디자인 자체가 문제라기보다,

“예쁜 다크테마 포폴 발표” 위에 “회사 프로젝트 내용”을 얹어 놓은 느낌
이라서, 자연스럽게 과제 느낌이 나는 거.

2. 진짜 핵심: “사업/업무 관점”이 약하고 “기능 소개” 비중이 너무 큼

지금 슬라이드 흐름을 보면:

Step1~6 프로세스

Graph API 연동 요약

WBS & 진척률

기능 모듈 상세

화면별 기능 명세

산출물 리스트

리스크 & 의사결정

향후 계획

대부분이:

“무슨 기능이 있다”

“어떤 화면이 있다”

“어디까지 개발했다”

에 초점이 맞춰져 있음.
회사 PM/팀장이 듣고 싶은 건 보통 이거야:

이 프로젝트 안 하면 회사가 어떤 손해/비효율이 계속 나는지

이걸 하면 어디가 얼마나 개선되고, 인력/시간/리스크가 얼마나 줄어드는지

돈/리소스/일정 관점에서 지금 상태가 어떤지

지금 의사결정을 안 해주면 뭐가 막히는지

지금 자료엔:

“지금 공지 업무가 얼마나 비효율적인지”

“연간 몇 건 발송, 그 중 몇 %가 잘못 나가거나 누락되는지”

“관리자가 엑셀/수동으로 하는 일 시간이 얼마나 드는지”

“이번 시스템으로 그걸 어느 정도 줄이는지”

이런 수치/예상효과가 거의 없음 → 그래서 “상용 프로젝트” 느낌이 아니라 “기능 잘 만든 토이 프로젝트” 느낌이 남.

3. 진짜로 “돈/리소스” 얘기를 넣으면 좋아지는 포인트

질문 그대로 답하면:

“돈에 대한 언급이 없나?”

응, 여기서 말하는 “돈”은 딱 두 가지야.

(1) 비용/리소스 측면

예시로 이런 내용이 한 슬라이드만 있어도 급 회사스러워져:

투입 리소스 요약

개발 인원: 1명 (착수 ~ 현재 7주, 약 N MD 사용)

추가 인프라: DEV 서버 1대 (기존 GW 인프라 활용, 신규 장비 없음)

MS Graph / 메일 API: 기존 조직 라이선스/플랫폼 내에서 사용(추가 라이선스 없음 or 검토 중)

예상/실제 일정 비교

계획 개발 완료: 2/20 → 현재 예상: 2/6 (2주 앞당김)

테스트/오픈 예정: 2/27 → 계획대로 진행 중

이 정도만 넣어도

“아, 일정/리소스 관리하면서 가고 있구나”
라는 메시지가 들어감.

(2) 정량 효과(= 돈으로 환산 가능한 것들)

예를 몇 개 잡아서 스토리 만들어볼 수 있어:

현재 공지 메일:

주간 평균 공지 메일: 10건

1건당 작성/수신자 선정/발송/추적에 약 30분 × 담당자 1명

연간 10건 × 52주 × 0.5시간 = 260시간

시스템 도입 후:

승인/발송/캘린더 등록 자동화로
1건당 수작업 시간 30분 → 10분 내로 감소(예상)
→ 연간 약 173시간 절감(약 1개월 인건비 수준)

추가로:

잘못 발송/누락 공지로 인한 장애 대응, 재안내, 문의 응대 감소
→ “장애 공지 누락 시 1건당 대응 2~3시간 소요” 같은 가정도 가능

완전 정확한 숫자가 아니라도 **“규모감”**만 보여주면

“이게 그냥 공지 보내는 툴이 아니라, 인력/리스크 절감 프로젝트구나”
라는 느낌이 확 살아남.

4. “조직/운영” 얘기가 부족함

지금은 화면/기능/프로세스 중심이고,
**“누가 이 시스템을 어떤 책임으로 쓸지”**가 안 보임.

예를 들어 이런 것들이 한 슬라이드 정도에 정리되면 좋음:

운영 주체

시스템 소유: IT개발팀

공지 등록 주체: 각 시스템/업무 담당자

최종 승인자: IT팀장 또는 지정 관리자

역할

공지 등록자: 본문 작성, 대상·일정 설정

승인자: 승인/반려, 긴급공지 예외 판단

시스템: 승인 이력 저장, 발송/캘린더 등록, History 집계

정책

공지 유형별 승인 필요/불필요

발송 전 최소 리드타임

히스토리/로그 보관 기간

이런 걸 넣으면,

“기능이 뭔지”보다 “실제 조직 안에서 어떻게 굴러가는지”가 보이기 때문에
회사 프로젝트 느낌이 훨씬 강해짐.

5. 리스크/의사결정 슬라이드도 “회사 관점”을 더 세게

지금 08 / 10 슬라이드는 구조는 좋아.
근데 살짝 “리스트 나열” 느낌이야.

조금만 더 회사 말투로 바꾸면 좋아져:

리스크를 ‘미해결 시 영향’으로 적기

예:

Graph API/메일 API 권한 및 토큰 발급 정보
→ ⇒ 미해결 시 DEV 환경에서는 테스트 가능하나, 운영 전환 일정 지연 가능

의사결정 항목은 “요청사항” 형태로

긴급공지(L3) 예외 정책

제안안 A: L3 중 시스템 장애 공지는 승인 생략

제안안 B: 승인자 1인 자동 승인

⇒ 중간발표 시 팀장님께 안 선택 요청

이렇게 **“오늘 회의에서 어떤 결정을 받고 싶은지”**를 명확히 하면,
발표가 ‘보고’가 아니라 진짜 ‘중간점검/의사결정’ 자리가 됨.
