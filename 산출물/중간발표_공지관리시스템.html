1. 프로젝트 목적과 범위

Why 중심

핵심 문제

  • 공지 메일이 분산되어 승인·이력 관리가 어렵다.
  • 발송 일정/대상 관리가 수작업이라 오류 가능성이 높다.
  • 캘린더 공유/공지 추적이 체계화되어 있지 않다.

기대 효과

  • 조직 표준 프로세스 정착 (승인·감사 추적)
  • 메일/캘린더 자동화로 업무 시간 절감
  • 발송 결과와 이력 조회로 운영 투명성 강화

포함 기능

  • 공지 등록/승인/발송/조회
  • 발송 스케줄러
  • 메일·캘린더 연동
  • 첨부파일 처리 및 송신 이력

제외 기능

  • 앱 환경 구축 의무 없음
  • 외부 시스템 운영/관리 기능 제외
범위 기준: 기존 사내 표준(GW/메일/캘린더)과의 일관성 유지

3. 아키텍처 및 기술 구조

표준 환경 준수

React SPA

공지/승인/발송 API, 스케줄러

PostgreSQL

public 스키마 · notice_xxx 모델

외부 연동

SSO, 메일 발송 API, 캘린더 API

표준 환경 준수 이유: 기존 조직의 인증 구조와 운영 정책과 일관성을 맞추기 위해 표준 환경을 따랐습니다.

핵심 연동 흐름

  • SSO 로그인 → 권한/부서 기반 접근 제어
  • 외부 연동은 앱 전용 토큰(client_credentials), 내부 API는 사용자 위임 토큰(authorization_code)
  • 메일 발송 API → 승인 완료 공지 자동 발송
  • 캘린더 등록 API → Outlook 일정 자동 생성
  • 첨부파일 처리 → 파일 서버 업로드 및 링크 관리
