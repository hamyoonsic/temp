API 요청(/api/… 같은 것들): 토큰으로 보호하면 되고, 하나하나 라우트 추가할 필요 없음
SPA 화면 라우트(/NoticeDashboard, /notices/… 같은 것들): 브라우저 주소창에서 직접 들어올 수 있으니, 이건 “index.html로 포워딩” 규칙이 필요

즉, “모든 요청을 토큰 있으면 통과”로만 퉁치면 SPA에서는 문제가 생겨요. 왜냐하면 화면 라우트는 주소창 GET 요청에 Authorization 헤더를 붙일 방법이 없어서요.

3) 그런데 “토큰이 있는 경우”라면 화면 라우트도 서버가 막아야 하나?

보통은 아니요.

화면 라우트는 서버가 굳이 막지 않고 index.html은 내려줌

대신 “대시보드 데이터 API”는 토큰 없으면 401

React는 토큰 없으면 로그인 페이지로 보내거나, “로그인 필요” 화면을 보여줌

이게 SPA에서 가장 흔한 보안 설계야:

HTML/JS는 공개

데이터 API는 보호

“화면 자체를 서버에서 막고 싶다”면 쿠키 기반(백엔드 세션)로 바꾸거나, 서버가 로그인 페이지로 redirect하는 구조가 필요해서 복잡도가 확 올라가.

4) 네 질문에 딱 맞는 답

“모든 API 요청에 대해 토큰이 있는 경우 컨트롤러 라우트에 주소를 하나하나 추가할 필요 없을까?”

✅ API 쪽은 필요 없게 만들 수 있고(보통 /api/**로 묶어서), 그게 정석이야.
⚠️ 하지만 SPA 화면 라우트는 토큰 여부와 무관하게(주소창 GET엔 토큰을 못 붙이니까)
index.html로 보내는 규칙이 필요하고, 이건 “정규식 포워딩”으로 하나하나 추가를 없앨 수 있어.

원하면, 네 현재 경로 체계가 /v1/api/**인지 /api/**인지에 맞춰서

Security에서 “API만 보호”로 바꾸는 버전

IndexController를 “확장자 없는 요청 자동 포워딩”으로 바꾸는 버전
둘 다 너 프로젝트에 맞는 코드로 바로 써줄게.
